# 实验记录模板填写指南

本说明对应文件：

```text
实验记录要求/实验记录模板.json
```

这个 JSON 模板的目标是：

1. 统一记录一次训练或一次评估
2. 兼容“训练 + 测试”和“只做评估”两种情况
3. 保证最后写 `report` 和做 `ppt` 时不会缺关键信息

## 1. 什么时候用这个模板

建议每一个独立实验都新建一份 JSON。

一个“独立实验”通常指：

1. 一次新的训练 run
2. 或者对一个已有 checkpoint 做一轮新的评估

例如：

1. `mobilenet_v3_small full` 训练一次
2. 对 `mobilenet_v3_small/full/best.pt` 跑一次 `HybridForensics`
3. 对 `clip/full/best.pt` 跑 `jpeg85`

都可以各自对应一份 JSON。

## 2. 最上面的基础字段怎么填

### `schema_version`

固定写：

```json
"schema_version": "v1"
```

先不要改。

### `record_type`

可选值建议只有两个：

1. `train_and_eval`
2. `eval_only`

含义：

1. `train_and_eval`
   这份记录既包含训练，也包含后续测试
2. `eval_only`
   这份记录只包含评估，不包含训练

### `experiment_id`

这是这份实验记录的唯一名字，建议写清楚：

```text
模型_训练规模_测试目标_版本
```

例如：

```text
mobilenet_full_hybrid_v1
resnet_small10000_test_v1
clip_full_jpeg85_v1
dual_v1_test_v1
```

### `run_id`

这是训练 run 的编号。  
如果只是单独评估，也可以填评估 run 名称。

例如：

```text
mobilenet_full_run01
clip_full_run02
hybrid_eval_run01
```

### `date`

建议用：

```text
2026-04-10
```

### `owner`

写负责这个实验的人名或缩写。

### `stage`

建议从下面选一个：

1. `baseline`
2. `robustness`
3. `hybridforensics`
4. `ablation`
5. `final`

### `tags`

这是一个数组，可以同时打多个标签。

例如：

```json
"tags": ["baseline", "hybridforensics", "mobilenet"]
```

### `include_in_report` 和 `include_in_ppt`

布尔值：

1. `true` 表示预计会写进最终材料
2. `false` 表示暂时只是参考结果

## 3. 复现相关字段怎么填

### `git_commit`

填这次实验对应的 commit id。  
如果当时没提交，也可以先留空，但建议后面补上。

### `script_path`

填实际运行的脚本，例如：

```text
baseline/train_mobilenetv3_small.py
baseline/evaluate_hybridforensics.py
```

### `command`

直接写完整命令，后面复现会非常有用。

例如：

```text
KMP_DUPLICATE_LIB_OK=TRUE python "baseline/evaluate_hybridforensics.py" --checkpoint "baseline/outputs/mobilenet_v3_small/full/best.pt"
```

### `seed`

如果没有特殊说明，就填：

```json
4210
```

### `device`

写这次实验实际使用的设备，例如：

```text
mps
cpu
cuda
```

## 4. `purpose`、`hypothesis`、`compare_to` 怎么填

这三个字段非常重要，因为它们决定你后面是否知道“这次实验是干嘛的”。

### `purpose`

写这次实验要回答的问题。

例如：

```text
Evaluate whether the full MobileNetV3-Small checkpoint generalizes to HybridForensics.
```

### `hypothesis`

写你在跑之前的预期。

例如：

```text
The model should perform well in-domain but drop significantly on cross-dataset evaluation.
```

### `compare_to`

写要和哪些已有结果比较，是一个数组。

例如：

```json
"compare_to": [
  "mobilenet_full_test_v1",
  "mobilenet_one_epoch_hybrid_v1"
]
```

## 5. `data` 对象怎么填

这一部分记录数据和预处理口径。

### `train_data_root` / `val_data_root` / `test_data_root`

训练实验就写对应目录。  
如果只是评估，没有训练，可以留空。

例如：

```json
"train_data_root": "data/hemg_processed",
"val_data_root": "data/hemg_processed",
"test_data_root": "data/hemg_processed"
```

### `benchmark_data_root`

如果有外部 benchmark，就写在这里，例如：

```json
"benchmark_data_root": "data/hybridforensics/raw"
```

### `train_split` / `val_split` / `test_split`

默认通常就是：

```json
"train_split": "train",
"val_split": "val",
"test_split": "test"
```

### `bias_control`

写 `true` 或 `false`。

表示：

1. 是否采用统一分辨率、统一 JPEG 保存等偏差控制口径

### `train_size_per_class`

建议写：

1. `"full"`
2. `"10000"`
3. `"5000"`

不要只写 `small`，后面容易忘。

### `input_resolution`

一般填：

```json
224
```

### `save_format`

例如：

```json
"jpeg"
```

### `preprocess_note`

简短写清楚预处理，例如：

```text
Upsample 32x32 to 256x256, center crop to 224x224, save as JPEG quality 95.
```

## 6. `model` 对象怎么填

### `model_name`

例如：

```text
mobilenet_v3_small
resnet18
clip
dual_stream_v1
```

### `method_family`

建议统一几种写法：

1. `baseline`
2. `forensic-only`
3. `dual-stream`
4. `ablation`

### `backbone`

例如：

```text
MobileNetV3-Small
ResNet18
CLIP ViT-B/32
```

### `pretrained`

是否使用预训练权重。

### `frozen_backbone`

是否冻结主干。

### `forensic_branch`

如果没有，就填：

```text
none
```

如果以后用了 `SRM` 或 `DCT`，就写对应名字。

### `fusion_method`

如果没有融合，就填：

```text
none
```

如果有融合，例如：

```text
concat_mlp
```

## 7. `training` 对象怎么填

这一块主要给训练实验用。

### `enabled`

1. 如果这份记录包含训练，填 `true`
2. 如果只是单独评估，填 `false`

### `output_dir`

例如：

```text
baseline/outputs/mobilenet_v3_small/full
```

### `checkpoint_path`

写最终用于测试的 checkpoint。

例如：

```text
baseline/outputs/mobilenet_v3_small/full/best.pt
```

### `loss_fn` / `optimizer` / `lr` / `weight_decay` / `batch_size` / `epochs` / `patience`

按实际训练配置填。

### `best_epoch`

填验证集最优轮数。

### `train_loss`

如果你只记录最后一个 epoch 的训练损失，就填最后值。

### `val_metrics`

填验证集最优或最后一次验证结果，建议统一写最优 checkpoint 对应结果。

### `in_domain_test_metrics`

填同分布测试集结果。

如果这次没有跑同分布测试，可以先留 `null`。

## 8. `evaluations` 数组怎么填

这部分最重要，因为最终 report 里的表几乎都来自这里。

一个元素代表一次评估结果。

### 常用的 `eval_type`

建议只用这几个：

1. `in_domain_test`
2. `robustness`
3. `cross_dataset`

### 每个评估都要填什么

至少要填：

1. `checkpoint_path`
2. `data_root`
3. `split`
4. `perturbation`
5. `metrics`
6. `num_samples`

### `perturbation`

如果没有扰动：

```json
"perturbation": {
  "type": "none",
  "jpeg_quality": null,
  "resize_setting": null
}
```

如果是 JPEG85：

```json
"perturbation": {
  "type": "jpeg",
  "jpeg_quality": 85,
  "resize_setting": null
}
```

如果是 resize：

```json
"perturbation": {
  "type": "resize",
  "jpeg_quality": null,
  "resize_setting": "128_to_224"
}
```

### `metrics`

建议统一填写：

1. `accuracy`
2. `auc`
3. `f1`
4. `loss`

### `skipped_files`

如果评估时有损坏图片被跳过，就把路径写进数组。  
如果没有，就填空数组：

```json
"skipped_files": []
```

## 9. `summary` 怎么填

这一部分是为了后面写 report 时能直接复制结论。

### `main_finding`

写这次实验最核心的一句话。

例如：

```text
The full MobileNetV3-Small checkpoint performs strongly on in-domain test but fails on HybridForensics.
```

### `failure_mode`

写主要失效形式，例如：

```text
cross_dataset_generalization_drop
jpeg_sensitivity
small_data_instability
```

### `is_abnormal`

如果结果异常高、异常低、和历史记录差别特别大，就写 `true`。

### `next_action`

写下一步最直接要做什么。

例如：

```text
Compare this result with the one_epoch_test checkpoint and verify label mapping.
```

### `report_sentence`

写一句几乎能直接放进 report 或 ppt 的话。

例如：

```text
Strong in-domain accuracy does not guarantee robust cross-dataset generalization.
```

## 10. 训练实验怎么用

如果这次是“训练 + 评估”，建议这样填：

1. `record_type = train_and_eval`
2. `training.enabled = true`
3. 填完整 `training`
4. 在 `evaluations` 里至少保留：
   - 标准测试
   - 一个鲁棒性测试
   - 一个 `HybridForensics` 测试

## 11. 只评估实验怎么用

如果这次只是对已有 checkpoint 做测试，建议这样填：

1. `record_type = eval_only`
2. `training.enabled = false`
3. `training.checkpoint_path` 仍然可以保留
4. 重点填写 `evaluations`
5. `purpose` 里写清楚这次只是在验证什么

## 12. 最低必填项

如果你赶时间，至少保证下面这些字段别空着：

1. `experiment_id`
2. `date`
3. `script_path`
4. `command`
5. `seed`
6. `model.model_name`
7. `training.checkpoint_path`
8. `evaluations[*].data_root`
9. `evaluations[*].metrics.auc`
10. `summary.main_finding`
11. `summary.report_sentence`

只要这些字段都在，最后大多数结果都还能追回来。

## 13. 一个最常见的错误

最常见的问题不是没跑实验，而是：

**只记了终端输出，没有记“这是哪个 checkpoint、和谁比较、说明什么”。**

所以你每次至少要多写 3 件事：

1. 这次用的是哪个 checkpoint
2. 这次和哪个结果比
3. 这次最想说明什么

## 14. 推荐习惯

每次跑完实验立刻做两件事：

1. 把终端指标填进 JSON
2. 先写 `summary.main_finding` 和 `summary.report_sentence`

因为最容易忘的不是分数，而是你当时为什么觉得这条结果重要。
