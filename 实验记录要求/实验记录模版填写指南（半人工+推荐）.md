# 实验记录模版填写指南（半人工 + 推荐）

## 最简单的方法 (来自萝卜头)

### 跑代码前，让AI参考 "实验记录要求/实验记录模板.json" 生成符合要求的脚本

### 跑代码后，让AI根据 "实验记录要求/"的所有文件整理好产生的output文件

本说明对应：

```text
实验记录要求/实验记录模板.json
```

这里的“半人工 + 推荐”是指：

1. `baseline/` 里的训练脚本和评估脚本已经会**自动生成一部分 JSON 字段**
2. 但和实验目的、结论、对照关系相关的内容，**仍然必须人工补** (推荐，不写也不要紧)
3. 最推荐的用法不是手工从头写完整 JSON，而是：
   先让脚本生成，再把 `需手动填写` 的地方补掉

## 1. 目前哪些脚本会按模板输出 JSON

### 训练脚本

1. `baseline/train_resnet18.py`
2. `baseline/train_mobilenetv3_small.py`
3. `baseline/train_clip.py`

这些脚本训练结束后，会在各自 `output_dir` 下生成：

```text
summary.json
```

而且已经是 `实验记录模板.json` 的结构。

### 评估脚本

1. `baseline/evaluate.py`
2. `baseline/evaluate_hybridforensics.py`

这两个脚本：

1. 终端会打印模板格式 JSON
2. 如果加 `--save-json`，也会把模板格式 JSON 保存到你指定路径

## 2. 推荐工作流

最推荐按下面流程用：

1. 跑训练脚本，拿到 `summary.json`
2. 打开这个 `summary.json`
3. 把所有 `需手动填写` 的字段补掉
4. 如果后续又对同一个 checkpoint 做 `jpeg` 或 `HybridForensics` 评估，就让评估脚本单独生成一份 JSON
5. 最后人工判断：
   这条记录是否写入 report / ppt

也就是说：

**训练类 JSON 负责保留 run 背景，评估类 JSON 负责保留某个 checkpoint 在某个测试设置上的结果。**

## 3. 哪些字段是脚本会自动填写的

下面这些字段，目前脚本一般能自动给出。

### 顶层基础字段

1. `schema_version`
2. `record_type`
3. `date`
4. `stage`
5. `tags`
6. `git_commit`
7. `script_path`
8. `command`
9. `seed`
10. `device`

### `data` 里的自动字段

训练脚本通常能自动给出：

1. `train_data_root`
2. `val_data_root`
3. `test_data_root`
4. `train_split`
5. `val_split`
6. `test_split`
7. `bias_control`
8. `train_size_per_class`
9. `input_resolution`
10. `save_format`

评估脚本通常能自动给出：

1. `test_data_root` 或 `benchmark_data_root`
2. `test_split`
3. `input_resolution`
4. `save_format`

### `model` 里的自动字段

当前脚本一般能自动给出：

1. `model_name`
2. `method_family`
3. `backbone`
4. `pretrained`
5. `frozen_backbone`
6. `forensic_branch`
7. `fusion_method`

### `training` 里的自动字段

训练脚本一般能自动给出：

1. `enabled`
2. `output_dir`
3. `checkpoint_path`
4. `loss_fn`
5. `optimizer`
6. `lr`
7. `weight_decay`
8. `batch_size`
9. `epochs`
10. `patience`
11. `best_epoch`
12. `train_loss`
13. `val_metrics`
14. `in_domain_test_metrics`

### `evaluations` 里的自动字段

评估结果相关字段目前基本都能自动给出：

1. `checkpoint_path`
2. `eval_type`
3. `data_root`
4. `split`
5. `perturbation.type`
6. `perturbation.jpeg_quality`
7. `perturbation.resize_setting`
8. `metrics.accuracy`
9. `metrics.auc`
10. `metrics.f1`
11. `metrics.loss`
12. `num_samples`
13. `skipped_files`
14. `note`

## 4. 哪些字段必须人工填写

下面这些字段，脚本目前不会自动推断，或者即使能猜也不可靠，所以统一保留成：

```text
需手动填写
```

### 一定要补的字段（萝卜头认为也不一定要补，看心情）

1. `owner`
2. `purpose`
3. `hypothesis`
4. `compare_to`
5. `summary.main_finding`
6. `summary.failure_mode`
7. `summary.next_action`
8. `summary.report_sentence`

### 强烈建议补的字段（不是萝卜头强烈建议的）

1. `include_in_report`
2. `include_in_ppt`
3. `data.preprocess_note`
4. `data.benchmark_data_root`（对训练脚本输出）
5. `data.train_size_per_class`（如果你们后面用了更复杂采样）
6. `evaluations[*].eval_id`

## 5. 为什么这些字段要人工填

因为这些内容不是“程序运行结果”，而是“实验语义”：

1. 你为什么跑这次实验
2. 你预期它说明什么
3. 你准备和谁比较
4. 这条结果能不能进最终材料
5. 它失败的原因你怎么解释

这些东西代码不知道，必须人来判断。

## 6. 各脚本特别需要注意的地方

### `train_resnet18.py`

自动会生成：

1. `resnet18` 的训练记录
2. 最优 `val` 指标
3. 标准 `test` 指标
4. 一条 `in_domain_test` 的 `evaluations`

你要手工补的重点：

1. 这次 `ResNet18` 是拿来和谁比较
2. 这是不是最终报告候选
3. 如果它表现差，是因为容量、泛化还是数据设置

### `train_mobilenetv3_small.py`

自动会生成：

1. `mobilenet_v3_small` 的训练记录
2. 最优 `val` 指标
3. 标准 `test` 指标
4. 一条 `in_domain_test` 的 `evaluations`

特别注意：

1. 默认数据根目录是 `data/hemg_processed`
2. 所以这条记录不一定和 `CIFAKE` 口径一致
3. 你必须手工确认这次实验到底属于哪条故事线

推荐手工补充：

1. `purpose`
   例如是否在验证更大训练集
2. `compare_to`
   例如和 `resnet18_full` 或 `clip_full`
3. `summary.report_sentence`

### `train_clip.py`

自动会生成：

1. `clip` 的训练记录
2. `clip_model_name`
3. 最优 `val` 和标准 `test`

特别注意：

1. 这个脚本默认是冻结主干，只训练线性头
2. 如果你们之后改过 CLIP 训练方式，不能直接沿用旧解释
3. `summary.failure_mode` 很值得认真写，因为它常常是 report 里的主 baseline

### `evaluate.py`

这个脚本适合：

1. 标准测试集评估
2. `jpeg95 / jpeg85 / jpeg75`
3. `test_resize`

自动会推断：

1. `eval_type`
2. `perturbation.type`
3. `jpeg_quality`

但要注意：

1. `test_resize` 的 `resize_setting` 现在会写成 `需手动填写`
2. 因为脚本只知道 split 叫 `test_resize`，不知道你具体怎么 resize 的

所以这里必须手工补：

1. `perturbation.resize_setting`
   例如 `128_to_224`
2. `summary.main_finding`
3. `compare_to`

### `evaluate_hybridforensics.py`

自动会给出：

1. `cross_dataset` 类型评估
2. `benchmark_data_root`
3. `num_samples`
4. `skipped_files`

特别注意：

1. 这个脚本输出的是**跨数据集结果**
2. 但 `training.enabled = false` 时，训练来源并不知道
3. 所以 `train_data_root`、`purpose`、`compare_to` 这些都要你手动补

最推荐补的内容：

1. 这个 checkpoint 是在哪个数据上训练的
2. 它的同分布结果是多少
3. 和哪个 benchmark 结果做比较

## 7. 目前脚本自动值里最容易误用的地方

下面几项虽然是自动填的，但你不能直接无脑拿去写 report。

### `bias_control = True`

当前脚本里默认就是 `True`，这更像是**项目约定**，不一定代表你这次实验真的严格检查过全部偏差控制流程。

所以：

1. 如果这次实验没有走统一预处理口径，记得手动改成 `false`
2. 或者至少在 `preprocess_note` 里说明真实情况

### `training.in_domain_test_metrics`

对于 `eval_only` 脚本来说，这个字段只是沿用了模板结构。
如果你跑的是 `HybridForensics`，它不是真的 “in-domain test”，只是脚本里暂时复用了这个位置。

所以：

1. 真正用于写作时，优先看 `evaluations`
2. 不要把 `evaluate_hybridforensics.py` 里的 `training.in_domain_test_metrics` 直接当同分布测试结果

### `command`

这个字段是从当前命令行自动抓的，通常可用。
但如果你是通过别的包装脚本调用，或者命令里省略了环境变量，它可能不够完整。

建议：

1. 重要实验最好人工检查一遍
2. 确认 `KMP_DUPLICATE_LIB_OK=TRUE` 这类环境变量有没有被记录进去

## 8. 最少要手工检查哪几处

如果你时间很紧，每份 JSON 至少检查下面 8 个地方：

1. `owner`
2. `purpose`
3. `compare_to`
4. `data.preprocess_note`
5. `include_in_report`
6. `summary.main_finding`
7. `summary.failure_mode`
8. `summary.report_sentence`

只要这 8 个地方认真补了，后面写 report 基本不会“只剩指标，没有解释”。

## 9. 推荐手工填写模板

你可以直接按下面这个口径补：

### `purpose`

```text
验证该模型在标准测试/压缩扰动/外部 benchmark 上的表现，并与已有 baseline 做比较。
```

### `hypothesis`

```text
模型在同分布测试上表现较好，但在更强扰动或跨数据集设置下会明显掉点。
```

### `compare_to`

```json
["resnet18_full_train_and_eval", "clip_full_train_and_eval"]
```

### `summary.main_finding`

```text
该结果表明模型在标准测试集上表现稳定，但在更严格设置下出现明显性能下降。
```

### `summary.failure_mode`

```text
cross_dataset_generalization_drop
```

可选写法还有：

1. `jpeg_sensitivity`
2. `small_data_instability`
3. `preprocess_dependency`

### `summary.next_action`

```text
继续补充 jpeg85、resize 和 HybridForensics 结果，并与 CLIP baseline 做对照。
```

### `summary.report_sentence`

```text
Strong in-domain performance does not guarantee robust generalization under perturbation or cross-dataset evaluation.
```

## 10. 哪些结果最值得优先补全

如果你们当前实验很多，不可能每条都认真整理，建议优先补下面几类：

1. 每个模型的 `full + test`
2. 每个核心模型的 `small-train + test`
3. 每个核心模型的 `jpeg85` 或 `jpeg75`
4. 每个核心模型的 `HybridForensics`

因为最后 report 和 ppt 最常用的就是这几组。

## 11. 最终建议

最推荐的习惯不是：

```text
跑完很多实验，最后一起补 JSON
```

而是：

```text
每跑完一个关键结果，立刻打开 JSON，把所有 “需手动填写” 的地方补掉
```

因为最容易丢的不是分数，而是：

1. 你当时为什么跑
2. 你想和谁比
3. 你觉得这个结果说明了什么

这三件事一旦拖久了，最后最难补。
