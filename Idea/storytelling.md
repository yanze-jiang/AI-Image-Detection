# report 里的故事怎么想

## 先定一句话主线

这份 report 最好的讲法，不是“我们做了一个 AI 图像检测模型”，而是：

**在资源受限条件下，单看同分布测试精度是不够的；我们先构建一个公平、轻量、可复现的检测框架，再用外部 benchmark 和扰动测试证明 baseline 的泛化不足，最后顺理成章地提出双流融合来解决这个问题。**

这句话很重要，因为它决定了整篇 report 的重心：

1. 重点不是追求一个最高分
2. 重点是说明“为什么这个问题值得这样做”
3. 重点是把你们的方法设计和实验结果串成因果链

## 这个项目最自然的叙事顺序

推荐按下面 5 步来讲：

### 1. 先讲现实约束，而不是先讲模型

开头不要一上来就说 `ResNet18`、`CLIP`、`SRM`。  
先讲一个更像研究问题的起点：

- AI 生成图像越来越逼真，检测很重要
- 但很多现有方案依赖大数据、大模型和高算力
- 课程项目里更现实的问题是：**在有限算力下，能不能做一个轻量、可复现、而且不容易被简单后处理击穿的检测器？**

这样一来，你们选择 `CIFAKE`、强调轻量化、做压缩测试、做外部 benchmark，都会显得非常自然。

一句话说，你们不是在做“更大的模型”，而是在做“更可信的轻量检测研究”。

### 2. 再讲为什么不能只看原始测试集分数

这里是 report 的第一个关键转折。

你们要告诉老师：

- 如果只在同一个数据集的标准测试集上看 `Accuracy`
- 很容易高估模型真实能力
- 因为模型可能学到的是格式差异、分辨率差异、数据集特有纹理
- 而不是真正稳定的伪造痕迹

所以你们专门做了两件事：

1. **偏差控制**
   统一分辨率、统一 JPEG 保存、控制明显伪线索
2. **更严格评估**
   除了标准测试，还看压缩/缩放扰动和 `HybridForensics` 外部 benchmark

这一步非常重要，因为它会把你们的实验设计从“跑 baseline”升级成“验证模型是否真的学对了东西”。

### 3. 然后讲 baseline 不是终点，而是发现问题的工具

你们的 baseline 不要写成“例行公事”。  
要写成：

**我们先用轻量 baseline 建立一个可比较的起点，并用它来观察模型到底在哪些场景下失效。**

这里可以这样组织：

- `MobileNetV3-Small`：代表轻量 CNN
- `ResNet18`：代表更经典的 CNN
- `CLIP + linear head`：代表更强的预训练语义特征

这样不是单纯堆模型，而是在构造一条比较线：

- 轻量 CNN 会不会更脆弱？
- 预训练语义特征会不会更稳？
- 小样本下谁更适合资源受限场景？

## 故事真正的“爆点”

整篇 report 最值得强调的结果，不一定是某个最高分，而是这个发现：

**模型在训练分布内可以表现不错，但一旦切到更真实的外部分布，性能明显下降。**

你们现在已经有一个很好的证据：

- `MobileNetV3-Small` 在 `HybridForensics` 上
- `Accuracy = 0.5231`
- `AUC = 0.5919`

这个结果的价值非常高，因为它天然回答了两个问题：

1. 为什么只看 `CIFAKE` 测试集是不够的
2. 为什么后面要做双流融合

也就是说，**不是因为我们想加一个更复杂的模块，所以提出双流；而是因为 baseline 已经暴露出泛化不足，所以我们才需要引入取证特征。**

这就是一个很完整的研究叙事闭环。

## 双流方法应该怎么被讲出来

双流方法不要讲成“我们又加了一条支路”。  
应该讲成：

**baseline 暴露出单一视觉表征不足以支持跨分布泛化，因此我们尝试把高层语义信息和底层取证信息结合起来。**

具体逻辑是：

1. 语义流负责看整体内容和结构
2. 取证流负责看残差、高频、压缩痕迹这类更底层的信号
3. 两者可能互补
4. 因此设计一个轻量双流融合模型，在不显著增加训练成本的前提下提升鲁棒性

这样写，方法部分就不是“结构介绍”，而是“为了解决前面发现的问题而提出的最小必要改进”。

## report 每一节真正要回答什么

### 1. Introduction

这一节不要写成大而空的背景综述。  
它只需要回答 3 个问题：

1. 为什么 AI 图像检测值得做
2. 为什么资源受限和鲁棒性是你们特别关心的点
3. 为什么只看同分布精度会有问题

结尾最好落到一句明确目标：

> We study whether a lightweight detector can remain effective under controlled preprocessing, post-processing perturbations, and cross-dataset evaluation.

### 2. Data and Protocol

这一节的核心不是介绍数据集百科，而是告诉读者：

**我们怎么把问题设置得更公平。**

重点写：

- 为什么选 `CIFAKE`
- 为什么保留 `Hemg` 作为补充训练入口
- 为什么引入 `HybridForensics`
- 为什么要做统一预处理和偏差控制
- 为什么主指标用 `AUC`

一句话概括这节：

**我们不是随便找数据训练，而是在尽量避免“模型靠伪线索取胜”。**

### 3. Baseline and Motivation

这节其实是 report 的承上启下部分。

你们要让读者看到：

- baseline 是必要的
- baseline 也确实能工作
- 但 baseline 的能力边界很明显

如果这里写得好，后面的方法改进就会很自然。

### 4. Method

这一节不要太长。  
因为你们真正的亮点不是复杂结构，而是“问题设置 + 针对性改进”。

建议写法：

- 先用 2-3 句回扣前文失效现象
- 再引出“语义流 + 取证流”
- 最后说明为什么这个设计是轻量的、合理的、可复现的

### 5. Experiments

实验部分不要按“模型一个一个报分数”来写。  
要按“问题一个一个被回答”来写。

推荐顺序：

1. 标准测试集表现如何
2. 小样本或轻量训练下谁更稳
3. 压缩/缩放扰动下谁掉得更少
4. 外部 benchmark 下谁更有泛化性

这样写出来会很像在回答研究问题，而不是贴结果表。

### 6. Conclusion

结论不要只写“我们的方法提升了 xx”。  
更好的写法是：

1. 同分布高分不代表真实泛化
2. 公平预处理和更严格评估是必要的
3. 双流融合如果能在扰动和外部分布上更稳，就说明底层取证特征确实有价值

最后再诚实写局限性：

- 数据来源还是有限
- 扰动类型还不够丰富
- 双流设计还比较初步

这样老师会觉得你们是清楚自己在做什么的。

## 你们这份 report 最好避免的讲法

下面几种讲法会让故事变弱：

1. 一上来就说“我们用了什么模型”
2. 把 `CIFAKE` 上的高分当成最终结论
3. 只强调 novelty，不解释为什么这个改进是被结果逼出来的
4. 实验部分只是堆表格，不解释每个实验回答了什么问题
5. 把双流说成万能方案，回避局限性

## 现在这个项目最推荐的主叙事版本

如果你们最后把双流结果也跑出来了，最推荐的版本是：

### 版本 A：完整研究故事

1. 资源受限条件下，需要轻量而可信的 AI 图像检测
2. 单看同分布测试精度不够，所以我们加入偏差控制、扰动测试和外部 benchmark
3. baseline 在原始分布内有效，但跨数据集泛化明显不足
4. 因此提出语义流 + 取证流的轻量双流融合
5. 实验表明双流在鲁棒性或外部泛化上更稳定

这会是最完整、最顺的 report。

如果最后双流结果还没完全跑完，也可以讲成：

### 版本 B：以“发现问题”为核心

1. 我们构建了一个轻量、可复现、偏差控制更严格的检测实验框架
2. baseline 在标准测试集上表现不错
3. 但在压缩扰动和 `HybridForensics` 上显著退化
4. 这说明当前轻量检测器仍存在明显的数据集依赖
5. 因此双流融合是一个有充分动机的后续改进方向

这个版本虽然没有最终改进结果那么强，但仍然是一个成立的课程项目故事，而且逻辑是完整的。

## 你写摘要时可以直接参考的口径

可以按这个意思来写：

> 本项目研究资源受限场景下的 AI 生成图像检测问题。我们基于 `CIFAKE` 构建轻量、可复现的实验框架，并通过统一预处理减少格式和分辨率偏差。在此基础上，我们评估了多个 baseline 在标准测试、后处理扰动和外部 benchmark 上的表现。结果表明，模型虽然能在同分布测试集上取得较好性能，但迁移到 `HybridForensics` 时明显下降，说明跨数据集泛化仍是核心挑战。基于这一发现，我们进一步采用语义流与取证流融合的思路，以提升模型在真实场景下的稳健性。

## 如果让我压成答辩时最该讲的 4 句

1. 我们关注的不是“能不能在单一数据集上刷高分”，而是轻量模型在更真实场景下是否可靠。
2. 因此我们不仅做标准测试，还做偏差控制、压缩扰动和外部 benchmark。
3. baseline 结果说明同分布高分不等于跨数据集泛化，`MobileNetV3-Small` 在 `HybridForensics` 上只有 `0.5919` 的 `AUC`。
4. 所以后续引入语义流和取证流的双流融合，不是为了堆复杂度，而是为了解决 baseline 暴露出的泛化不足。

## 最后给你的一个判断标准

你们写 report 的时候，每一页都可以问自己一句：

**这一页是在回答“我们为什么这么做”，还是只是在描述“我们做了什么”？**

如果前者更多，这个故事就会顺。  
如果后者更多，report 就容易变成流水账。
